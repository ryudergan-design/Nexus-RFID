Pilar 01 · Fundação

🏛️ Arquitetura de Software

Arquitetura não é apenas "como organizar pastas". É a decisão de como o sistema vai crescer, onde vai falhar, e o custo de cada mudança futura. Toda escolha arquitetural tem trade-offs.

Padrões Arquiteturais

  • Monolito Modular — Ideal para início. Tudo junto, mas bem separado internamente
  • Microserviços — Serviços independentes. Alta complexidade operacional
  • BFF (Backend for Frontend) — API específica por tipo de cliente
  • Event-Driven — Componentes comunicam via eventos assíncronos
  • Serverless — Funções gerenciadas pelo provedor cloud
  • Hexagonal (Ports & Adapters) — Isola o core de dependências externas

Princípios Essenciais

  • SOLID — Single Responsibility, Open/Closed, Liskov, Interface Segregation, Dependency Inversion
  • DRY — Don't Repeat Yourself
  • YAGNI — You Aren't Gonna Need It (não over-engineer)
  • Separation of Concerns — Cada módulo faz uma coisa
  • Loose Coupling — Componentes pouco dependentes entre si
  • High Cohesion — Coisas relacionadas ficam juntas

Camadas de uma Aplicação

  • Apresentação — Controllers, Routes, Middlewares
  • Aplicação — Use Cases, Services, Commands/Queries
  • Domínio — Entities, Value Objects, Domain Events
  • Infraestrutura — Repositórios, ORM, APIs externas
  • Transversal — Logs, Auth, Error handling

Para Vibe Coding

  • Defina a estrutura de pastas antes de codar
  • Escreva o README arquitetural primeiro — explique para a IA como o projeto funciona
  • Use convenções explícitas — a IA segue padrões claros
  • Separe domínio da infra — facilita substituir dependências
  • ADRs (Architecture Decision Records) — documente decisões importantes

Pilar 02 · Persistência

🗄️ Banco de Dados

O banco de dados é onde a realidade do seu negócio persiste. Modelagem errada é difícil de desfazer. Entender quando usar SQL vs NoSQL, como indexar e como migrar dados com segurança é fundamental.

Tipo Exemplos Quando Usar Trade-offs
SQL Relacional PostgreSQL, MySQL, SQLite Dados estruturados, relacionamentos complexos, transações ACID Escala vertical, schema rígido — mas robusto e consistente
Document Store MongoDB, Firestore Dados semi-estruturados, esquema flexível, alto volume de escrita Sem JOINs nativos, eventual consistency
Key-Value / Cache Redis, Memcached Sessões, cache, filas, rate limiting, pub/sub Volátil por padrão, não é fonte de verdade
Search Engine Elasticsearch, Meilisearch Busca full-text, autocomplete, faceted search Complexo de manter, usar em conjunto com DB principal
Time-series InfluxDB, TimescaleDB Métricas, IoT, logs com timestamp Especializado — não substituir o DB principal

Boas Práticas de Modelagem

  • Normalize para consistência — evite dados duplicados
  • Desnormalize para performance — quando as queries exigem
  • Use UUIDs ou ULIDs como PKs em sistemas distribuídos
  • Soft delete — não delete dados, marque como deletado
  • Timestamps em tudo — created_at, updated_at, deleted_at
  • Não armazene senhas — apenas hashes (bcrypt, argon2)

Migrations e Evolução

  • Nunca altere o schema direto em produção
  • Use migrations versionadas — Prisma, Flyway, Liquibase
  • Backward compatible — adicione colunas com default antes de remover antigas
  • Rollback plan — toda migration precisa ter um down
  • Teste migrations em staging antes de produção
  • Blue-green para grandes mudanças de schema

Para Vibe Coding: Sempre peça para a IA gerar o schema do banco ANTES do código. Defina o modelo de dados, relacionamentos e índices primeiro. Uma mudança de schema depois custa 10x mais do que planejar bem antes.

Pilar 03 · Lógica de Negócio

⚙️ Backend

O backend é o centro nervoso da aplicação. Aqui vive a lógica de negócio, o processamento, as integrações e a exposição de APIs. Qualidade aqui se traduz diretamente em confiabilidade.

REQUEST
Client envia requisição HTTP → Load Balancer distribui → chega no servidor
MIDDLEWARE
Auth (verifica JWT/session) → Rate Limiter → Validator (schema do body) → Logger
CONTROLLER
Extrai dados da request, chama o Use Case / Service correspondente, trata erros
SERVICE
Lógica de negócio — valida regras, chama repositórios, emite eventos, coordena operações
DATA LAYER
Repository acessa DB → Cache Redis → Queue para jobs assíncronos
RESPONSE
Serializa dados → aplica DTO/Schema de saída → retorna com status code correto

APIs bem projetadas

  • REST — GET /users, POST /users, PUT /users/:id
  • Status codes corretos — 200, 201, 400, 401, 403, 404, 422, 500
  • Versionamento — /api/v1/... para não quebrar clientes
  • Paginação — cursor ou offset, nunca retorne tudo
  • Filtros e ordenação — ?sort=created_at&order=desc
  • Documentação — Swagger/OpenAPI gerado automaticamente

Patterns críticos

  • Repository Pattern — abstrai o acesso a dados
  • Dependency Injection — inverte controle de dependências
  • Command/Query Separation — operações de leitura ≠ escrita
  • Circuit Breaker — protege de falhas em cascata
  • Idempotência — mesma request → mesmo resultado
  • Dead Letter Queue — jobs que falharam não são perdidos
Pilar 04 · Interface

🎨 Frontend

O frontend é o único pilar que o usuário toca diretamente. Performance, acessibilidade e UX determinam se o produto é adotado ou abandonado.

Rendering Strategies

  • CSR (Client-Side Rendering) — SPA clássico, React puro. Ruim para SEO
  • SSR (Server-Side Rendering) — Next.js. HTML gerado no servidor por request
  • SSG (Static Site Generation) — Build-time. Ultra rápido, ideal para conteúdo estático
  • ISR (Incremental Static Regen.) — SSG com revalidação em background
  • Streaming SSR — envia HTML em chunks, percepção de velocidade

State Management

  • Local State — useState/useReducer para estado do componente
  • Server State — TanStack Query/SWR para dados da API (cache, revalidação)
  • Global State — Zustand, Jotai para estado compartilhado mínimo
  • URL State — filtros e paginação na URL (shareable, back/forward funciona)
  • Form State — React Hook Form + Zod para validação

Web Vitals (Core)

  • LCP — Largest Contentful Paint. Deve ser < 2.5s
  • FID/INP — First Input Delay / Interaction to Next Paint. < 200ms
  • CLS — Cumulative Layout Shift. < 0.1
  • TTFB — Time to First Byte. Impacto do servidor
  • Bundle size — code splitting, lazy loading, tree shaking

Design System

  • Tokens de design — cores, tipografia, espaçamento como variáveis
  • Componentes reutilizáveis — Button, Input, Modal, Table, etc.
  • Storybook — documentação viva dos componentes
  • Acessibilidade (A11y) — aria-*, contraste, navegação por teclado
  • Dark mode — CSS custom properties com prefers-color-scheme
Pilar 05 · Proteção

🔒 Segurança

Segurança não é uma fase do projeto — é uma camada transversal. Os ataques mais comuns são os mais simples, e a maioria das brechas vem de negligência, não de técnicas sofisticadas.

✗
SQL Injection
Nunca concatene strings em queries. Use prepared statements e ORM sempre.
✗
XSS (Cross-Site Scripting)
Sanitize inputs, use CSP headers, nunca confie em dados do usuário no DOM.
✗
CSRF
CSRF tokens em forms, SameSite cookies, verificar Origin header.
✗
Broken Auth
JWT com expiração curta, refresh tokens rotativos, invalidação no logout.
✗
Secrets Expostos
Nunca commite .env. Use Secret Manager (AWS, Doppler, Vault).
✗
Rate Limiting
Limite requests por IP/usuário. Protege de brute force e DDoS básico.
✗
IDOR (Insecure Direct Object Ref.)
Sempre verifique se o usuário tem permissão no recurso que está acessando.
✗
Dependências Vulneráveis
npm audit, Snyk, Dependabot automático no CI/CD.

Regra de ouro de segurança: "Nunca confie em nada que veio de fora do seu servidor." — inputs do usuário, headers HTTP, callbacks de webhooks, dados de APIs externas. Valide, sanitize e autentique tudo.

Pilares 06 e 08 · Confiabilidade

🧪 Testes e QA

Testes são a documentação viva do comportamento esperado. QA é o processo sistemático de garantir que o sistema funciona como deveria — e continua funcionando após cada mudança.

UNIT TESTS
Testam funções isoladas. Rápidos, sem dependências externas. Use mocks. Cobrem lógica de negócio pura. Tools: Jest, Vitest, pytest
INTEGRATION
Testam módulos juntos. Controller + Service + DB real (ou in-memory). Verificam que as partes se conectam. Tools: Supertest, TestContainers
E2E TESTS
Simulam o usuário real. Abre browser, clica, preenche forms, verifica resultado. Mais lentos, mais caros. Tools: Playwright, Cypress
CONTRACT
Garante que API não quebrou. Consumer-driven contracts com Pact. Essencial em microserviços
PERFORMANCE
Testa carga e stress. Quantas requests simultâneas o sistema aguenta? Tools: k6, Artillery, Locust

Coverage não é tudo. 100% de coverage com testes ruins é pior do que 60% de coverage com testes que testam comportamento real. Foque no comportamento, não nas linhas.

Hierarquia de Testes (Akita)

  • Testes unitários — isolamento extremo, diretórios isolados por worker para paralelização sem interferência
  • Pipeline CI rápida — lint + segurança + testes em ~22 segundos. Roda em cada commit
  • Pipeline de integração E2E — com dados reais via rsync de produção, chamadas reais a APIs com cache file-based (DevCache)
  • Preflight / smoke test — valida estrutura, contagens mínimas e marcadores obrigatórios antes do pipeline principal
  • Preview visual manual — o único que captura bugs estéticos que automação não detecta

Testes de Integração em Monorepo

  • Contratos implícitos são o maior risco — múltiplos apps comunicando via filesystem/Markdown com frontmatter YAML
  • Testes unitários não validam contratos entre apps — só o pipeline E2E detecta divergências
  • Commits atômicos no monorepo — alteram ambos os lados do contrato (gerador e consumidor) no mesmo commit
  • rsync de dados de produção — testa com dados reais, não fixtures
  • Flag FORCE para bust de cache e SKIP_EXISTING para gerar só o que mudou
Pilar 07 · Operação

🚀 DevOps e CI/CD

O software só cria valor quando está rodando em produção. DevOps é a disciplina de colocar código em produção com rapidez, segurança e confiabilidade — e mantê-lo funcionando.

Pipeline CI/CD Típico

  • 1. Push → trigger automático no GitHub/GitLab
  • 2. Lint + Type Check → falha rápido em erros óbvios
  • 3. Unit Tests → validação da lógica
  • 4. Build → compila e empacota a aplicação
  • 5. Integration Tests → testa com dependências reais
  • 6. Security Scan → SAST, dependências vulneráveis
  • 7. Deploy Staging → ambiente espelho do produção
  • 8. E2E Tests → valida o fluxo completo
  • 9. Deploy Production → blue-green ou canary

Observabilidade (3 Pilares)

  • Logs — registro estruturado (JSON) de eventos. Ferramentas: Datadog, CloudWatch, Loki
  • Métricas — números ao longo do tempo: latência, error rate, CPU, memória. Ferramentas: Prometheus + Grafana
  • Traces — rastreia uma request por todos os serviços. Ferramentas: Jaeger, Zipkin, OpenTelemetry
  • Alertas — notificação quando métricas saem do normal (PagerDuty, Opsgenie)
  • SLOs/SLAs — defina e monitore metas de disponibilidade (99.9% uptime = 8.7h downtime/ano)

Containerização

  • Docker — empacota a app com suas dependências. Mesmo comportamento em dev e prod
  • Docker Compose — orquestra múltiplos serviços localmente
  • Kubernetes (k8s) — orquestração em produção, auto-scaling, self-healing
  • Dockerfile multi-stage — build menor e mais seguro
  • Never run as root — segurança em containers

IaC (Infra como Código)

  • Terraform — provisiona cloud infra de forma declarativa
  • Pulumi — IaC com linguagens de programação reais
  • AWS CDK / CloudFormation — específico para AWS
  • Ansible — configuração de servidores
  • GitOps — infraestrutura gerenciada via Git (ArgoCD, Flux)
Metodologia · Como Usar Tudo Isso

⚡ Vibe Coding Profissional

Vibe coding com IA não é "deixar a IA fazer tudo". É saber orquestrar a IA com contexto rico e decisões arquiteturais conscientes. A qualidade do output é proporcional à qualidade do input.

Fase 1 · Planejamento (Você)

  • Escreva o IDEA.md / spec primeiro — requisitos em linguagem natural antes de qualquer código
  • Defina a arquitetura antes de escrever código
  • Escreva o schema do banco primeiro
  • Faça benchmarks antes de decidir — valide suposições técnicas com dados reais (benchmark-driven development)
  • Escolha o stack com critério (não com hype). Prototipe em linguagem mais fácil antes de otimizar

Fase 2 · Contexto para IA

  • CLAUDE.md / AGENTS.md — mantenha um arquivo com convenções, stack e contexto do projeto. A IA lê isso a cada sessão
  • Mostre a estrutura de pastas existente antes de pedir código
  • Dê exemplos de código do mesmo projeto para a IA seguir o padrão
  • Especifique o stack exato (versões, bibliotecas)
  • Peça testes junto com o código — sempre. A IA não ignora essa prática quando você exige

Fase 3 · Review (Você como Arquiteto)

  • Você é o arquiteto, a IA é o dev sênior — revise toda decisão estrutural
  • Nunca comite código da IA sem revisar — especialmente commits volumosos
  • Verifique segurança com ferramentas — Brakeman, Snyk, bundler-audit no CI
  • Cheque over-engineering — a IA tende a adicionar complexidade desnecessária
  • Entenda o código — se não entendeu, não está pronto para produção

Fase 4 · Commits e Iteração

  • Commits pequenos e descritivos — evite "bug fix" sem contexto. Facilita git bisect quando algo quebra
  • CI em cada commit — lint + testes rodando em ~22s. Se demorar mais, quebre em jobs paralelos
  • Small releases constantes — tudo pronto para produção a qualquer momento (XP)
  • Refatore continuamente — não deixe débito técnico acumular. Testes dão segurança para isso
  • Documente decisões — ADRs e CLAUDE.md. A IA (e você no futuro) agradece

📖 Lições do Mundo Real

Insights extraídos de projetos reais construídos com Vibe Coding intensivo — do zero à produção em dias, com qualidade profissional. Padrões que se repetiram em múltiplos projetos.

LIÇÃO 01

A IA é um dev sênior motivado — não um arquiteto

O que a IA faz bem

  • Gerar boilerplate rapidamente seguindo padrões estabelecidos
  • Escrever testes — e faz isso melhor que a maioria dos humanos quando exigido
  • Refatorar código com instruções claras e contexto suficiente
  • Identificar bugs óbvios e sugerir correções de segurança
  • Internacionalização, rebranding e tarefas repetitivas de alto volume
  • Tool calling — usar ripgrep, grep, LSPs, compiladores para validar o próprio código

Onde a IA falha

  • Contexto global do projeto — não enxerga o todo sem você fornecer
  • Over-engineering — tende a adicionar complexidade desnecessária (Opus, Kimi)
  • Decisões arquiteturais — simplistas, exigem revisão humana obrigatória
  • Edge cases sutis — race conditions, bugs de encoding, path handling cross-platform
  • Segurança profunda — Brakeman captura o que a IA deixa passar
  • Perda de contexto em conversas longas — CLAUDE.md resolve parcialmente

Analogia do Akita: "A IA age como um estagiário motivado — executa bem o que você pede, mas não substitui o desenvolvedor sênior que define o que deve ser feito, revisa as decisões e detecta o que foi ignorado." Diferentes modelos têm personalidades diferentes: alguns são prolixos (Opus, Kimi), outros mais diretos (GPT Codex), e cada um deixa passar bugs diferentes.

LIÇÃO 03

CI rodando em cada commit é inegociável

Pipeline CI de referência (Rails)

  • RuboCop / ESLint — padrão de código, falha rápida
  • Bundler-Audit — vulnerabilidades em dependências
  • Brakeman — análise estática de segurança (SAST)
  • Suíte completa de testes — todo o conjunto em ~22 segundos
  • SimpleCov — relatório de cobertura com threshold mínimo
  • GitHub Actions Matrix — builds e testes em múltiplos OS (Linux, macOS, Windows)

O papel do teste regressivo

  • Todo bug corrigido ganha um teste — sem exceções. Evita reincidência
  • Git bisect — commits pequenos permitem isolar regressões rapidamente
  • Pipeline de integração separada do CI — roda com dados reais via rsync de produção
  • Preview visual manual — captura bugs que automação não detecta
  • Health checks — banco de dados, serviços externos, antes de rodar
  • DevCache — cache local de chamadas à API para testes sem custo repetido
LIÇÃO 04

Segurança em 7 camadas — não é paranoia, é profissionalismo

O que uma auditoria real cobre

  • Rate limiting agressivo + IP banning (Rack::Attack)
  • Proteção contra race conditions — updates atômicos no banco (SELECT FOR UPDATE)
  • Sanitização de nomes de arquivos — path traversal, caracteres especiais
  • Detecção server-side de MIME type — não confie no Content-Type do cliente
  • TOTP com proteção contra replay attack
  • CSP com nonces aleatórios por request (não valor estático)
  • Open redirect prevention — valide destinos antes de redirecionar

Segurança de dados em repouso

  • Criptografia no ActiveRecord/ORM — emails e dados sensíveis criptografados no banco
  • AES-256 para dados em repouso em produção
  • Algoritmo determinístico para campos criptografados que precisam de unicidade (índices)
  • Quota de disco por usuário — protege contra abuse de armazenamento
  • Limpeza automática assíncrona — jobs para remover dados expirados
  • Invites únicos e atômicos — prevenção de race condition no registro
LIÇÃO 05

Resiliência e recuperação de falhas são features, não extras

Padrões de resiliência observados

  • Cancelamento cooperativo (AtomicBool em Rust) — app se recupera de crashes sem perder progresso
  • Scanning incremental — nunca reprocesse o que já foi feito. Fingerprint SHA-256 para detectar arquivos movidos
  • Cache persistente — retoma onde parou após interrupção
  • Modo read-only para diretórios scaneados — preserva dados do usuário
  • Migrações formais de banco — nunca altera schema manualmente. Garante upgrade seguro

Envio de email confiável (at-most-once)

  • Estado individual por email — tabela EmailDelivery com claiming atômico
  • Retry só em erros transitórios — não reenvia em bounced, rejected ou unknown
  • Backoff polinomial — evita flood em falhas temporárias do SES
  • Job de recuperação de stale — emails presos em "sending" por +30min voltam para fila
  • List-Unsubscribe (RFC 8058) — compliance e proteção de reputação no Gmail
  • Webhooks de reconciliação — fecha o ciclo via SNS para bounce/complaint

Web Scraping robusto (hierarquia de confiabilidade)

  • 1. API oficial — sempre a primeira opção
  • 2. RSS/Atom feed — estável, leve, semântico
  • 3. HTML server-rendered — Nokogiri, parsing direto
  • 4. SPA com Headless Browser — Ferrum + Chromium via CDP
  • 5. Sites anti-bot (último recurso) — injeção de scripts antes da página, mask de propriedades JS
  • Não dependa de seletores CSS — use heurísticas estruturais (tamanho de URL, agrupamento de links)

Deploy com Kamal (zero-downtime)

  • Kamal — deploy Rails em VPS próprio sem Kubernetes
  • SQLite em produção — com WAL mode para concorrência. Simples e eficiente para projetos solo/small team
  • Cloudflare Tunnel — SSL, DDoS protection e roteamento sem abrir portas no servidor
  • Docker multi-stage — imagem mínima e segura (não roda como root)
  • Deploy em 2 minutos — hotfix crítico de anti-spam do Gmail deployado em 120 segundos
